02

설계 결정 (Engineering Decisions)

단일 교사가 운영하는 무예산 환경에서, 다수 학생의 동시 접속·거래를 견디기 위해 내린 선택들. 각 항목은 마주한 문제 → 택한 방법 → 그 근거 순으로 정리했다. 항목을 펼쳐 상세를 볼 수 있다.

Concurrency

동시성과 데이터 정합성

자산 변동 트랜잭션의 직렬화LockService.getScriptLock() · waitLock(10000)
문제여러 학생이 같은 순간 거래하면, "잔액을 읽고 → 계산하고 → 다시 쓰는" 사이에 다른 처리가 끼어들어 자산이 어긋날 수 있다(race condition).
선택기부·경매 낙찰·P2P 송금·예금 가입 등 자산을 바꾸는 모든 함수를 getScriptLock()으로 감싸 직렬화했다. 락 획득은 최대 10초 대기하고, 실패하면 예외 대신 "다른 처리 중입니다"를 반환해 사용자가 재시도하게 한다.
근거GAS는 멀티 인스턴스로 동시 실행되므로 시트 읽기·쓰기 사이의 원자성이 보장되지 않는다. 락으로 임계 구간을 보호하되, 무한 대기로 사용자를 묶지 않도록 타임아웃과 친화적 메시지를 함께 두었다.
중첩 락(nested lock) 회피_consumeSuperPassInline_()
문제경매 낙찰 처리는 이미 락을 보유한 상태인데, 그 안에서 슈퍼패스(우선입찰권)를 차감하는 함수가 또 락을 잡으면 교착(deadlock) 위험이 있다.
선택슈퍼패스 차감용 내부 함수에는 의도적으로 락을 두지 않고, 호출하는 쪽이 보유한 락을 그대로 신뢰하도록 설계했다. 함수명과 주석에 "이미 락 보유 컨텍스트"임을 명시했다.
근거하나의 트랜잭션 안에서는 단일 락만 유지하는 것이 안전하다. 재진입 가능한 락이 없는 GAS 환경에서 중첩 획득은 잠재적 교착이므로, 락의 소유 책임을 호출 계층으로 일원화했다.
핵심 거래와 부가 기능의 실패 분리try/catch 격리
문제낙찰(핵심) 처리 중 슈퍼패스 차감이나 로그 기록(부가)에서 오류가 나면, 정작 중요한 거래까지 함께 실패할 수 있다.
선택부가 작업을 별도 try/catch로 격리했다. 슈퍼패스 차감에 실패해도 낙찰은 정상 처리하고 경고 메시지만 반환한다. 대화 로그 기록 실패도 AI 응답을 막지 않는다.
근거실패의 영향 범위를 좁히는 것이 견고성의 핵심이다. 사용자에게 가장 중요한 결과(거래 성립)를 부차적 기능의 오류로부터 보호했다.
멱등성(idempotency) — 중복 처리 방지보상 수령 · 평점 · 트리거 등록
문제학생이 보상 버튼을 두 번 누르거나, 같은 거래에 평점을 또 매기거나, 트리거를 중복 등록하면 자산이 두 번 지급되는 등 부작용이 생긴다.
선택호감도 보상은 히스토리 메모에 CLAIMED:캐릭터:STAGE3 플래그를 남겨 재수령을 차단한다. P2P 평점은 송금자만·1회만 가능하도록 검증한다. 백업·만기 트리거는 등록 전 동일 핸들러 존재 여부를 확인한다.
근거같은 요청이 여러 번 들어와도 결과가 한 번 적용된 것과 같아야 한다. 별도 상태 테이블 없이 기존 로그에 플래그를 심어 가볍게 멱등성을 확보했다.
Performance

응답 속도 최적화

낙찰당 9.9초 → 4초: 무거운 집계의 지연 처리executeAuctionSold() / getTodayAuctionResults()
문제경매는 한 회차에 수십 건이 빠르게 낙찰된다. 그런데 매 낙찰마다 전체 랭킹을 다시 계산(~1.4초)하고 학생 전체를 Firebase에 동기화(~2.5초)하니, 건당 약 9.9초가 걸려 진행이 끊겼다.
선택데이터 정확성에 필요한 기록(자산 차감·거래 로그·낙찰가)은 낙찰 즉시 시트에 쓰되, 표시 갱신용 집계(랭킹·Firebase)는 매 건마다 하지 않고 경매 종료 시 한 번에 일괄 처리하도록 분리했다. 건당 약 4초로 단축됐다.
근거"즉시 정확해야 하는 것"과 "잠시 후 최신화되어도 되는 것"을 구분하는 것이 핵심이었다. 어드민 입찰표의 잔액은 클라이언트에서 즉시 갱신되고 서버가 매번 시트에서 재검증하므로, 집계를 미뤄도 정합성은 깨지지 않는다.
부분 동기화: Firebase HTTP 요청 22회 → 2회syncOneStudentToFirebase()
문제P2P 송금은 당사자 2명의 자산만 바뀌는데, 기존엔 전체 학생을 Firebase에 다시 올려 매번 22번의 HTTP 요청이 발생했다.
선택전체 동기화(syncAllStudentsToFirebase)와 단건 동기화(syncOneStudentToFirebase)를 분리하고, 거래에는 당사자 2명만 동기화했다. 요청이 22회에서 2회로 줄었다.
근거변경된 데이터만 전송하는 것이 네트워크 비용의 기본이다. 외부 API 호출은 GAS에서 가장 비싼 연산 중 하나이므로 호출 횟수 자체를 줄였다.
시트 I/O 최소화 + 단계별 진단 타이머getValues()/setValues() 일괄 · _mark()
문제GAS에서 시트를 한 셀씩 읽고 쓰는 것은 매우 느리다. 또 "어디서 느린지" 모르면 최적화할 수 없다.
선택전 구간에서 getDataRange().getValues()로 한 번에 읽어 메모리에서 처리한 뒤 setValues()로 일괄 기록한다. 낙찰가 기록 시에도 전체 범위 대신 카테고리·상세명 두 열만 읽어 매칭한다. 낙찰 처리에는 단계별 소요 시간(ms)을 측정하는 진단 타이머를 내장해 병목을 추적한다.
근거시트 호출 횟수가 곧 속도다. 읽기·쓰기를 배치로 묶고, 추측이 아니라 실측으로 병목을 찾아 위의 9.9→4초 개선에 도달했다.
콜드 스타트 완화: 수업 시간 워밍업 트리거keepAlive() · everyMinutes(1)
문제GAS 인스턴스는 한동안 호출이 없으면 잠들어, 학생이 오랜만에 접속할 때 첫 응답이 느려진다(cold start).
선택1분마다 빈 함수 keepAlive()를 실행하는 워밍업 트리거를 두어 수업 시간 동안 인스턴스를 활성 상태로 유지한다. 수업이 없는 주말·방학에는 트리거를 끄도록 메뉴에 안내한다.
근거서버를 상시 켜둘 수 없는 GAS 환경에서 응답 지연을 줄이는 실용적 절충안이다. 불필요한 시간대에는 꺼서 실행 할당량 낭비를 막는다.
기술 부채의 명시적 문서화recordCreditScoreSnapshot() 주석
문제신용점수 계산은 학생 1명당 시트 5개를 읽는다. 23명이면 약 110번의 시트 읽기가 순차 실행되며, 데이터가 누적될수록 GAS 6분 실행 제한에 가까워질 수 있다.
선택현재 규모에서는 문제가 없으므로 섣불리 최적화하지 않되, 코드 주석에 위험 시점(학년 말 누적 데이터, 30명 이상)과 개선 방향(시트를 함수 바깥에서 한 번 읽어 파라미터로 전달)을 명시해 두었다.
근거불필요한 조기 최적화는 복잡도만 키운다. 대신 부채를 인지하고 문서화해, 임계점에 도달했을 때 즉시 대응할 수 있게 했다.
Resilience

상태 관리와 운영 안정성

경매 상태의 이중 저장: Cache 우선 + Properties 백업CacheService + PropertiesService
문제실시간 경매의 상태(진행 중/낙찰/유찰, 종료 시각)는 빠르게 읽혀야 하는데, GAS의 CacheService는 빠르지만 휘발성이라 만료·소실될 수 있다.
선택상태를 캐시(TTL 6시간)에 우선 저장하되, 영속적인 PropertiesService에도 동시에 백업한다. 읽을 때 캐시에 없으면 Properties에서 복구한 뒤 다시 캐싱한다.
근거속도(캐시)와 영속성(Properties)은 트레이드오프다. 둘을 계층으로 결합해 평소엔 빠르고, 캐시가 날아가도 상태를 잃지 않는 구조를 만들었다.
매일 자동 백업과 30개 순환 보관runDailyBackup() · _cleanOldBackups(folder, 30)
문제교사의 실수나 데이터 손상에 대비한 복구 수단이 필요하다. 그러나 백업을 무한히 쌓으면 드라이브 용량을 잠식한다.
선택매일 자정 트리거로 스프레드시트 전체를 드라이브 전용 폴더에 복사한다. 백업이 30개를 넘으면 가장 오래된 것부터 자동 삭제하는 보관 정책(retention)을 두었다. 백업 실패는 try/catch로 격리해 본 시스템에 영향을 주지 않는다.
근거데이터를 신뢰할 수 있어야 학급 자산이라는 민감한 기록을 다룰 수 있다. 자동화 + 순환 보관으로 사람의 개입 없이 복구 지점을 유지한다.
시간 기반 자동화 트리거 묶음Time-driven Triggers
문제예금 만기 지급, 브랜드가치 일별 기록, 신용점수 주간 스냅샷, 비상사태 자동 해제 같은 정기 작업을 사람이 매번 챙길 수 없다.
선택매일 12:30 예금 만기 처리, 매일 23시 브랜드가치 기록, 비상사태는 종료 시각에 맞춘 일회성 트리거(autoEndEmergency)로 자동 해제한다. 트리거 등록 시 항상 동일 핸들러 중복을 먼저 제거한다.
근거운영을 자동화해야 교사가 교육에 집중할 수 있다. 중복 등록 방지로 트리거가 누적·중복 실행되는 사고를 막았다.
Security

접근 제어와 입력 보안

학생 무권한 접속을 위한 실행 주체 설계executeAs: USER_DEPLOYING · ANYONE_ANONYMOUS
문제학생에게 구글 계정 로그인이나 시트 공유 권한을 요구하면 진입장벽이 너무 높고, 시트에 직접 접근 권한을 주면 데이터가 위험하다.
선택웹앱을 배포자(교사) 권한으로 실행(USER_DEPLOYING)하고 익명 접근(ANYONE_ANONYMOUS)을 허용한다. 학생은 시트 권한 없이 URL만으로 접속하고, 신원은 시트에 저장된 자체 비밀번호로 검증한다.
근거초등학생이 계정 없이 즉시 사용할 수 있어야 한다는 요구와, 원본 데이터를 보호해야 한다는 요구를 동시에 만족시키는 구성이다. 모든 데이터 접근이 서버 측 함수를 거치므로 학생은 시트를 직접 보지 못한다.
입력 검증·살균과 비밀 분리_sanitizeString() · _validateStudentName() · PropertiesService
문제익명 접근이 가능하므로, 학생 이름·비밀번호 같은 입력에 악의적 문자열이 섞일 수 있다. 마스터 비밀번호를 코드에 하드코딩하면 노출 위험이 있다.
선택이름은 길이(20자)와 특수문자를 차단하고, 출력 문자열은 HTML 엔티티로 이스케이프한다. 마스터 비밀번호는 코드가 아닌 PropertiesService의 스크립트 속성에 분리 저장한다.
근거신뢰할 수 없는 입력은 검증하고, 비밀은 코드와 분리한다는 보안의 기본 원칙을 익명 웹앱 환경에 맞게 적용했다.
AI Engineering

AI 캐릭터 — 비용·안전·맥락

2중 안전장치: 로컬 필터(비용 0) → AI 문맥 판정_hasBannedWord_() → _callClaude_()
문제초등학생 대상 AI 대화는 욕설·부적절 발화를 걸러야 한다. 그러나 모든 메시지를 AI로 검사하면 비용이 들고, 명백한 욕설까지 API를 부르는 것은 낭비다.
선택1차로 로컬 금지어 필터를 API 호출 전에 돌려(비용 0) 명백한 욕설은 즉시 차단한다. 통과한 메시지만 2차로 AI가 문맥을 판정해 무례함의 수위(none/mild/severe)를 가린다. 심각(severe)은 경고 누적 없이 즉시 대화를 잠그고, 교사만 해제할 수 있다.
근거싸고 확실한 검사(로컬)를 먼저, 비싸고 정교한 검사(AI)를 나중에 배치하는 단계적 방어다. 비용과 안전을 동시에 잡는다. 잡담용으로는 경량 모델(claude-haiku)을 선택해 운영 비용을 낮췄다.
학생 경제 맥락 주입형 프롬프트_buildEconomySummary_() · 히스토리 10턴
문제일반 챗봇은 학생의 실제 상황을 몰라 일반론만 말한다. "저축하라" 같은 조언이 이 학급의 규칙(연말 종료, 등교일만 활동)과 맞지 않을 수 있다.
선택대화마다 그 학생의 자산·활동 요약과 이 세계의 규칙을 시스템 프롬프트에 주입하고, 직전 대화 10턴을 함께 전달해 맥락을 유지한다. 호감도 단계에 따라 노출되는 정보를 게이트한다.
근거맥락 없는 조언은 교육적 가치가 낮다. 학생의 데이터를 컨텍스트로 넣어, AI가 그 학생에게 맞는 한 가지를 짚도록 했다.
03

도메인 로직의 정교함

실제 경제·금융의 원리를 단순 흉내가 아니라 작동하는 규칙으로 구현했다. 모든 수치는 코드에 정의된 값이다.

실질 순자산 기반 랭킹

자산 순위를 단순 보유액이 아니라 실질 순자산으로 계산한다. 진행 중 예금은 자산으로, 대출 잔액은 부채로 반영한다.

랭킹 자산 = 보유 자산 + 진행중 예금 원금 − 대출 잔액

1,000점 4트랙 신용점수

신뢰·투자·기여·준법을 분리 평가한다. 모든 트랙이 학급 내 상대평가이며, 데이터가 없을 땐 기본 50점을 준다.

트랙A 신뢰도(400, 내 BV÷최고 BV) + 투자성실(200) + 사회기여(200) + 규범준수(200)

무죄 추정: 의심과 확정의 분리

이상거래는 '의심'(자동 감지)과 '최종 적발'(교사 확정)을 구분한다. 신용점수 차감은 최종 적발일 때만 일어난다. 수호대는 1차 판정, 교사는 최종 판단을 맡는다.

이상거래 소급 플래그

동일인 간 거래가 하루 3회째에 도달하는 순간, 이미 '정상'으로 기록된 1·2번째 거래까지 소급해서 이상거래로 표시한다. 단일 거래 $2,000 이상도 이상거래로 본다.

신용 연동 대출 심사

신용등급이 한도와 금리를 동시에 결정한다. 한도는 자산의 0~120%, 주간 금리는 3~12%. 진행 중 대출이 있으면 신규 대출을 막고, 상환 기한은 실행 후 28일이다.

복수 예금상품과 판매 중단

여러 예금상품을 동시 운영하며, 상품 '종료'는 신규 가입만 차단하고 진행 중 예금은 만기까지 유지한다. 학생별 누적 한도로 한도 우회를 막는다. 만기는 매일 12:30 트리거가 자동 지급한다.

거시 충격: 비상사태 시스템

자산 동결(사용 가능액을 보유의 N%로 제한)과 하이퍼인플레이션(경매 시작가 2배) 같은 거시 변수를 교사가 발동할 수 있다. 종료 시각에 맞춘 일회성 트리거가 자동 해제한다.

길드 점수와 기여 상한(cap)

월간 점수는 자산 성장(0.50)·미션 참여(0.15)·세션 참석(0.10)·미션(0.25)의 가중합이다. 개인의 브랜드가치 기여에 길드 평균의 2배라는 상한을 두어 소수 독점을 방지한다.

단일 진실 공급원(SSOT)

P2P 거래 자격(금 광석 티어)을 판정할 때 숫자(20000)를 직접 비교하지 않고 _calcTier()의 티어 순서 인덱스로 비교한다. 코드 주석에 "숫자 직접 수정 금지 — 이중 관리 방지"를 명시했다.

지니계수·로렌츠 곡선 직접 구현

자산 분포의 불평등을 표준 공식으로 계산한다. 로렌츠 곡선 아래 면적을 사다리꼴 적분으로 구해 지니계수를 산출하고, 수호대 대시보드에 시각화한다.
